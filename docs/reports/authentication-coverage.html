Gaps & recommendations

Business paths not yet closed, ordered by how much regression risk they carry.

High
Injection / XSS payloads in the Login form aren't tested
CLAUDE.md's cross-cutting scenarios require that script/SQL payloads in form fields render as literal text. Search and Registration already cover this; Login's e-mail/password fields don't. Add a case submitting a payload like ' OR 1=1-- / <script> and assert it's rejected as an ordinary no-match credential, not reflected unescaped.
Medium
Post-login redirect back to the originally-requested page is unverified
When a guest is bounced to Login from a protected page (Wishlist, Checkout), it's untested whether a successful login returns them to that page or always lands on My Account. During exploration this behaved inconsistently depending on prior navigation — worth pinning down with an explicit test rather than leaving it assumed.
Medium
TC-ACCOUNT-013 (guest wishlist → login redirect) has no automated test
Documented in TC-ACCOUNT.md and referenced by Workflow 8, but no spec exercises it. Natural to pair with the redirect-back gap above in one test.
Medium
Cart contents after logout are not asserted
CLAUDE.md explicitly lists “session/cart behavior after logout” as a scenario. TC-LOGIN-015 confirms the session ends, but doesn't check whether cart line items survive the logout (the Logout page's own copy claims they're saved and restored on next login).
Low
Login email is not tested for case-insensitivity
Logging in with the registered email in a different letter case (e.g. Name@Example.com vs. name@example.com) is a common real-world path and is currently unverified either way.
Out of scope
Completing a password reset via the emailed link isn't automated
TC-FORGOT-016 verifies the request is accepted and the confirmation message shown; actually following the reset link and setting a new password requires inbox access this suite doesn't have. Flagging as a manual-test note rather than a closable gap.
